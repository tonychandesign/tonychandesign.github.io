Context

From 2014 to 2018, Intuit’s design technology team created the Intuit Prototyping Framework, which stored end-to-end prototypes used for qualitative user testing.

Testing high fidelity, end-to-end prototypes allows us to gain authentic qualitative insights from customers before we invest in building new innovations for our products.

Users are more likely to behave realistically when testing prototypes that look like “live” software. This belief is also backed by many research organizations around the world, like the Nielsen Norman Group. Most prototyping tools do not use real components or real data well, often resulting in research studies where participants must imagine they are in a specific scenario, which leads to less reliable insights than if they were able to use their real information.

Intuit Prototyping Framework (2014-2019)

The Intuit Prototyping Framework gave the business a cheap way of collecting realistic design feedback in a low-risk environment before the handoff to software development.

Platform pitfalls

There were a lot of issues with this outdated framework. After interviewing several designers, we created a list of requirements for a successful prototyping platform.

Desired Characteristics

Competitors

Before spending the time to build an entire platform, there was still the option of using an external prototyping tool. I reached out to product specialists from a handful of companies to see if they could fulfill our needs. I used this opportunity to learn about limitations, essential features, and common design patterns used by such tools.

Competitors' features

Most of these tools couldn’t consume production design system components. This meant that designers would have to maintain the components alongside our existing design system. Updating multiple systems whenever there was a change is difficult to maintain and prone to version control inconsistencies so this was a deal breaker.

FramerX checked most of our boxes so I worked with their engineering team to set up a proof of concept. Unfortunately, they were unable to load our design system onto their app and creating complex flows actually required coding experience.

Common design patterns

User Tests

Aside from myself, there was one other design technologist working on this project. We set an aggressive timeline to build a proof of concept in 6 months so that designers could use the tool for user testing before the tax season started. We prioritized development over design initially. The intent was to get a functional application working then observe how designers used the tool. Once we got real usage data, we could start ironing out the user experience.

This is not a design tool replacement for Sketch or Figma. It’s a prototyping tool specifically made for building and testing high fidelity ideas without using costly development resources.

Main priorities:

  • Transition the tech stack from AngularJS to ReactJS
  • WYSIWYG editor which used production design system components
  • Routing functionality to build flows
  • Persistent user data

WYSIWYG Editor

Beta Video Demo

Designers could link their editor built pages to more complex pages that required development resources. This reduced the bottleneck that our team had on the design process because we were no longer responsible for updating every single page. The end users testing the prototypes wouldn’t be able to tell the difference between a coded page versus an editor built page.

Prototype architecture

We released the pilot to 8 designers who used the tool to create prototypes for a month. They reported bugs and gave us feedback on what to improve before we released the tool to the greater design organization.

Designer Feedback:

  • Pages needed tags for organization and searchability.
  • It was difficult to find images, it’s too unorganized.
  • Needed image browser in the editor.
  • Wanted to see the pages in a flow.

Research Data

Findings

Friction often occurs in the high fidelity phase. The team pivots often and needs to collaborate closely in this stage; members lose visibility over decisions made, resulting in a conflicting source of truth.

Intuit's design process

Journey map

Since the teams move so rapidly, research findings behind certain iterations are lost or forgotten. Intuit teams needed a historical research database which links to prototypes and shows the reasoning behind pivots.

These questions came up often in product meetings:

  • “Why was this pivot done in Quarter 2?”
  • “Did we try this before?”
  • “Which designs failed? Why?”

Opportunity: Prior iterations & rationale get buried because teams use multiple tools to test ideas so it takes longer to make calculated decisions

For the usability test, I had the participants try to recreate a simple flow using the editor. We needed more feedback to know how to pivot for the next iteration.

Usability task

Feedback:

  • Users thought it was difficult to use the JSON editor to update component content.
  • Users wanted to see where the page sits in the flow.
  • Users wanted to edit multiple pages at once, instead of one at a time.
  • Users were confused by inconsistent inline editing vs right panel JSON editing.
  • Users felt that copying and pasting urls to replace a component’s images was tedious. They wanted a “choose image” button.
  • Users wanted the ability to comment.

Component JSON editor

I hosted a design sprint and had my teammates draw out how they thought content should be organized for a prototyping tool. Most of the sketches were bucketed into 2 categories. The first was a subdirectory layout that used folders, similar to Box or Google Drive. The second design pattern was to list all of the pages and include a robust search/tagging system to navigate through the content.

Subdirectory approach

Robust search approach

I converted the sketches into two low fidelity prototypes then tested each version randomly amongst the research participants.

Robust search approach

Subdirectory approach

Feedback:

  • Users preferred the filter approach.
  • Users had mixed opinions about using subdirectories because only the organizer would know how the subdirectories are organized. It also required a lot of maintenance.
  • Users didn’t care about the specifics of shared content but did want a private workspace for WIP designs.
  • Users didn’t understand the need for a pages section. They felt that a page should always be within the prototype so it’s relative to the context.
  • Some users felt that the filter didn’t need an entire sidebar. A good filter can be achieved with just a search bar.
Next Steps

At this point we aligned on a few goals:

  • Users need to collaborate on prototypes asynchronously.
  • Users need real time saving- similar to Figma’s collaborative auto save.
  • Users need to duplicate prototypes easily to ensure previous iterations aren’t lost.
  • Users need to edit multiple pages in a prototype at once.
  • Users need consistent inline editing within the editor.
  • Users need the content to be organized by prototypes, instead of pages.
  • Users need a robust search and tagging interface.
  • Users need logic based navigation.

We began researching into different WYSIWYG editors that fit our needs and came across CraftJS. It’s an open source website builder which offers a comprehensive API around inline text editing and nested containers.
